# ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ) // CYBER-STREAM-CORE
**Продукт:** Предиктивный движок киберспортивной аналитики реального времени (Dota 2 / CS2) с AI-модулем оценки драфтов.  
**Целевой стек платформы:** Node.js 24, NestJS 11, TypeScript (nodenext), Next.js 16 (Turbopack), PostgreSQL 15, Redis 7, BullMQ, RabbitMQ, Prisma 7, Anthropic/Claude SDK, Docker, Helm.

---

## 1. БИЗНЕС-КОНТЕКСТ И ЦЕЛЬ
В условиях киберспортивного live-стриминга задержка данных или обработка дубликатов пакетов ведет к финансовым потерям (в B2B-линиях букмекеров) и ошибкам аналитики (для тренерского штаба). 

**Цель проекта:** Разработать отказоустойчивое ядро (Core Engine) монорепозитория, способное принимать плотный, хаотичный поток live-событий матча, фильтровать его на лету от сетевых аномалий, транслировать в реальном времени на клиентский пульт управления через WebSockets и генерировать предиктивный AI-анализ драфтов без деградации производительности СУБД.

---

## 2. АРХИТЕКТУРНЫЕ ТРЕБОВАНИЯ К КОМПОНЕНТАМ (МОНОРЕПОЗИТОРИЙ)

### 2.1 apps/tracker-service (Бэкенд-ядро на NestJS)
*   **API Gateway & WebSocket Сервер:** Сервер должен поднимать WebSocket Gateway для удержания активных соединений с клиентами и трансляции игровых тиков в реальном времени (Real-time Feed).
*   **Слой валидации (DTO):** Жесткая фильтрация входящих пакетов через `ValidationPipe`. Защита от XSS-инъекций и обфусцированного мусора в live-стриме через кастомные регулярные выражения.
*   **Слой Идемпотентности (Redis):** Реализация распределенного защитного замка через `SET NX` с префиксом `message:UUID`. Повторные дубликаты пакетов от внешних скрейперов/букмекеров должны отбиваться на входе со статусом `409 Conflict`, не долетая до СУБД.
*   **Атомарный слой (PostgreSQL + Prisma):** Запись пакетов событий матча в базу данных через механизм `$transaction` с гарантией Rollback при сетевых сбоях инфраструктуры.

### 2.2 apps/live-worker (Фоновые пайплайны на BullMQ + Redis)
*   **Буферизация нагрузки:** Очереди на базе BullMQ + Redis для изоляции парсинга и фоновой AI-обработки. 
*   **Транспорт очередей:** Интеграция RabbitMQ-буфера для асинхронного batch-логирования игровых событий матча. Консьюмер должен работать в режиме **Manual Ack** с отстойником ядовитых сообщений (Dead Letter Exchange — DLX) для предотвращения эффекта "раздвоения мозгов" и потери live-данных.

### 2.3 client/control-plane (Пульт управления на Next.js 16)
*   **Интерфейс:** Одностраничное SPA-приложение (`'use client'`) в киберспортивной стилистике холдинга Team Spirit (Dark-mode, минимализм, высокая читаемость метрик).
*   **Интерактив:** Реализация живого логгера событий. Кнопка «Сымулировать кибер-атаку дубликатами пакетов Steam» должна через `Promise.all` отправлять пачку одновременных сетевых запросов для визуальной демонстрации работы Redis-замка в реальном времени.

## 3. ТРЕБОВАНИЯ К ДАННЫМ И ОПТИМИЗАЦИИ (СУБД)
*   **Схема базы данных:** Описание связей «Матчи — Логи игровых событий (Game Events)». Использование кастомного локального вывода клиента Prisma (`output = "client"`) для изоляции контекстов монорепозитория.
*   **Highload-оптимизация:** Принудительное проектирование составных B-Tree индексов на уровне СУБД по полям `[matchId, timestamp]`. Выборка исторических графиков золота/опыта конкретного матча должна происходить за наносекунды, исключая Full Table Scan.
*   **Таймзоны и Локали:** Полная изоляция хранения дат — запись в Postgres строго в UTC (`DateTime`). Вывод на клиентскую сторону через нативный системный инструмент Node.js/Браузера `Intl.DateTimeFormat` на основе часового пояса конкретного аналитика (Париж, Сиэтл, Шанхай).

---

## 4. AI-МОДУЛЬ (Интеграция Claude SDK)
*   Реализация сервиса интеграции с **Anthropic/Claude API SDK** .
*   Метод должен принимать текущий драфт героев (выбранные персонажи матча) и генерировать предиктивный текстовый вердикт (Win Rate, сильные/слабые стороны пика команды) для вывода на панель мониторинга.

---

## 5. INFRASTRUCTURE & OBSERVABILITY (DevOps)
*   **Контейнеризация:** Стерильная сборка Dockerfile через Multi-stage (отсечение dev-зависимостей, оптимизация веса Linux Alpine образа).
*   **Оркестрация:** Написание масштабируемых локальных манифестов `docker-compose.yml` и подготовка верхнеуровневых Umbrella Helm-чартов для деплоя платформы в Kubernetes-кластер одной командой.
*   **Логирование:** Внедрение структурированного логирования ключевых бизнес-событий (решения `allow / deny` по трансляции коэффициентов, фиксация сбоев RabbitMQ) для последующей агрегации в стек Grafana/Loki.
