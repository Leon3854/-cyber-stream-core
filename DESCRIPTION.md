

# Главная идея: 
Создать "мозг" для киберспортивной аналитики, который должен переваривать огромный поток данных с матчей (Dota 2, CS2) в реальном времени и работать без остановки, даже если сверху сыпется мусор или одни и те же данные по сто раз.

---

## 1. С чем мы воюем? (Бизнес-контекст)
*   **Проблема:** Во время матча данные летят с бешеной скоростью. Если данные приходят с опозданием или дублируются (одно и то же событие пришло дважды), это катастрофа. Для букмекеров это потеря денег, для тренеров — неверные выводы по игре.
*   **Главная цель:** Сделать ядро системы, которое:
    *   Принимает этот дикий поток данных.
    *   Мгновенно отсеивает мусор и повторы.
    *   Показывает всё на экране аналитикам в реальном времени.
    *   Использует ИИ, чтобы предсказывать шансы на победу на стадии выбора героев (драфта), и не даёт базе данных "лечь" под нагрузкой.

---

## 2. Архитектура (или "Что строим")

Строим монорепозиторий (все части проекта в одной папке), разделённый на три главных приложения:

*   **`apps/tracker-service` (Диспетчер и привратник):**
    *   Это центральный сервер. Он ловит данные по WebSockets (как постоянная прямая линия связи) и тут же рассылает их всем, кто смотрит.
    *   **Защита:** Он проверяет каждое входящее сообщение на грязь (XSS-атаки, мусор) через жёсткие правила (DTO + ValidationPipe).
    *   **Защита от повторов (самое главное!):** Использует Redis. Каждому сообщению присваивается уникальный ID. Сервер запоминает этот ID в Redis с командой "запомнить, если такого ещё не было". Если сообщение с таким ID уже пришло, сервер просто говорит "409 Conflict" и выкидывает его. Это как штамп на руке в клубе — второй раз по одному штампу не пустят.
    *   **Запись в базу:** Записывает важные события в PostgreSQL атомарно (операция "всё или ничего" через `$transaction`). Если на полпути что-то сломалось, база откатывается, чтобы не было "битых" данных.

*   **`apps/live-worker` (Чернорабочие на фоне):**
    *   Чтобы основной сервер не тормозил, раздавая данные, тяжёлую работу (парсинг, ИИ-анализ) он скидывает в очередь.
    *   **BullMQ (на базе Redis):** Это как список задач, который фоновая программа разбирает по порядку.
    *   **RabbitMQ:** Ещё один, более навороченный, список задач для массовой записи логов в базу. Здесь важно **Manual Ack и Dead Letter Exchange**. Это значит: выполнение задачи, и только потом говорит: "Всё ок, удали задачу". Если задача сломала (стала "ядовитой"), она уходит в специальный "отстойник" для разбора, а не зацикливается и не убивает всю систему ("предотвращение раздвоения мозгов").
 
*   **`client/control-plane` (Пульт управления):**
    *   Это сайт (интерфейс) в тёмном, минималистичном стиле Team Spirit. Делается на Next.js 16, работает как SPA.
    *   **Фича:** Будет кнопка, которая запускает **симуляцию атаки**. Она отправит кучу одинаковых запросов одновременно (`Promise.all`), чтобы показать, как круто работает защита от дублей из пункта про `tracker-service`.

#### 3. Работа с данными

*   **Схема:** Нужно описать в коде связи: у одного Матча есть много Игровых Событий.
*   **Скорость запросов:** Чтобы график золота/опыта за матч строился мгновенно, в базе данных нужно создать особый индекс по полям `[matchId, timestamp]`. Это как алфавитный указатель в книге, только для базы данных — она не перечитывает все страницы, а сразу идёт в нужное место.
*   **Время:** Храни все даты в базе строго по Гринвичу (UTC). А показывай их аналитику уже в его местном времени (в Париже, Сиэтле или Шанхае), используя встроенный `Intl.DateTimeFormat`.

---

#### 4. ИИ-модуль (Мозг для предсказаний)
Ты подключаешь **Claude (Anthropic SDK)**. Делаешь сервис, который получает на вход список уже выбранных героев (драфт) и просит ИИ выдать вердикт текстом: "Примерный шанс на победу — 65%, пик силён в ранней игре, но слаб в поздней". Этот текст потом показывается на пульте управления.

---

#### 5. Инфраструктура (Упаковка и развёртывание)

*   **Docker Multi-stage:** Ты создаёшь легковесный "контейнер" для своего приложения. Первый этап — собираешь код (где есть всё для разработки), второй этап — берёшь только готовый результат и кладёшь в чистую и маленькую операционную систему Alpine. Это делает образ приложения лёгким и безопасным.
*   **Запуск:** `docker-compose.yml` для запуска на компе разработчика и Helm-чарты для запуска в промышленном Kubernetes-кластере (как гигантский оркестр из контейнеров).
*   **Логирование:** Все важные действия (пропустили пакет, сбой в RabbitMQ) должны писаться в логи в структурированном виде (JSON), чтобы потом Grafana/Loki могли красиво показать их на дашбордах.
