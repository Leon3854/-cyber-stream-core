

# Главная идея: 
Создать "мозг" для киберспортивной аналитики, который должен переваривать огромный поток данных с матчей (Dota 2, CS2) в реальном времени и работать без остановки, даже если сверху сыпется мусор или одни и те же данные по сто раз.

---

## 1. С чем мы воюем? (Бизнес-контекст)
*   **Проблема:** Во время матча данные летят с бешеной скоростью. Если данные приходят с опозданием или дублируются (одно и то же событие пришло дважды), это катастрофа. Для букмекеров это потеря денег, для тренеров — неверные выводы по игре.
*   **Главная цель:** Сделать ядро системы, которое:
    *   Принимает этот дикий поток данных.
    *   Мгновенно отсеивает мусор и повторы.
    *   Показывает всё на экране аналитикам в реальном времени.
    *   Использует ИИ, чтобы предсказывать шансы на победу на стадии выбора героев (драфта), и не даёт базе данных "лечь" под нагрузкой.

---

## 2. Архитектура (или "Что строим")

Строим монорепозиторий (все части проекта в одной папке), разделённый на три главных приложения:

*   **`apps/tracker-service` (Диспетчер и привратник):**
    *   Это центральный сервер. Он ловит данные по WebSockets (как постоянная прямая линия связи) и тут же рассылает их всем, кто смотрит.
    *   **Защита:** Он проверяет каждое входящее сообщение на грязь (XSS-атаки, мусор) через жёсткие правила (DTO + ValidationPipe).
    *   **Защита от повторов (самое главное!):** Использует Redis. Каждому сообщению присваивается уникальный ID. Сервер запоминает этот ID в Redis с командой "запомнить, если такого ещё не было". Если сообщение с таким ID уже пришло, сервер просто говорит "409 Conflict" и выкидывает его. Это как штамп на руке в клубе — второй раз по одному штампу не пустят.
    *   **Запись в базу:** Записывает важные события в PostgreSQL атомарно (операция "всё или ничего" через `$transaction`). Если на полпути что-то сломалось, база откатывается, чтобы не было "битых" данных.

*   **`apps/live-worker` (Чернорабочие на фоне):**
    *   Чтобы основной сервер не тормозил, раздавая данные, тяжёлую работу (парсинг, ИИ-анализ) он скидывает в очередь.
    *   **BullMQ (на базе Redis):** Это как список задач, который фоновая программа разбирает по порядку.
    *   **RabbitMQ:** Ещё один, более навороченный, список задач для массовой записи логов в базу. Здесь важно **Manual Ack и Dead Letter Exchange**. Это значит: выполнение задачи, и только потом говорит: "Всё ок, удали задачу". Если задача сломала (стала "ядовитой"), она уходит в специальный "отстойник" для разбора, а не зацикливается и не убивает всю систему ("предотвращение раздвоения мозгов").
 
*   **`client/control-plane` (Пульт управления):**
    *   Это сайт (интерфейс) в тёмном, минималистичном стиле Team Spirit. Делается на Next.js 16, работает как SPA.
    *   **Фича:** Будет кнопка, которая запускает **симуляцию атаки**. Она отправит кучу одинаковых запросов одновременно (`Promise.all`), чтобы показать, как круто работает защита от дублей из пункта про `tracker-service`.

#### 3. Работа с данными

*   **Схема:** Нужно описать в коде связи: у одного Матча есть много Игровых Событий.
*   **Скорость запросов:** Чтобы график золота/опыта за матч строился мгновенно, в базе данных нужно создать особый индекс по полям `[matchId, timestamp]`. Это как алфавитный указатель в книге, только для базы данных — она не перечитывает все страницы, а сразу идёт в нужное место.
*   **Время:** Храни все даты в базе строго по Гринвичу (UTC). А показывай их аналитику уже в его местном времени (в Париже, Сиэтле или Шанхае), используя встроенный `Intl.DateTimeFormat`.

---

#### 4. ИИ-модуль (Мозг для предсказаний)
Ты подключаешь **Claude (Anthropic SDK)**. Делаешь сервис, который получает на вход список уже выбранных героев (драфт) и просит ИИ выдать вердикт текстом: "Примерный шанс на победу — 65%, пик силён в ранней игре, но слаб в поздней". Этот текст потом показывается на пульте управления.

---

#### 5. Инфраструктура (Упаковка и развёртывание)

*   **Docker Multi-stage:** Ты создаёшь легковесный "контейнер" для своего приложения. Первый этап — собираешь код (где есть всё для разработки), второй этап — берёшь только готовый результат и кладёшь в чистую и маленькую операционную систему Alpine. Это делает образ приложения лёгким и безопасным.
*   **Запуск:** `docker-compose.yml` для запуска на компе разработчика и Helm-чарты для запуска в промышленном Kubernetes-кластере (как гигантский оркестр из контейнеров).
*   **Логирование:** Все важные действия (пропустили пакет, сбой в RabbitMQ) должны писаться в логи в структурированном виде (JSON), чтобы потом Grafana/Loki могли красиво показать их на дашбордах.

## Как это работает в реальности

Представь себе экосистему: есть поставщики данных, твой сервис и конечные потребители. В жизни это выглядит как слоеный пирог из сервисов.

### Слой 1: Источники данных (то, чего нет в ТЗ)

В реальности никто не ждет, что данные сами прилетят. Нужны **адаптеры-коннекторы**:

*   **GSI (Game State Integration)** для Dota 2/CS2 — официальный протокол Valve для получения данных из игры. Твой сервис должен уметь парсить GSI-пакеты в реальном времени.
*   **Steam Web API** — для получения данных о матчах, игроках, героях.
*   **Webhook-приемники от букмекерских контор** — многие B2B-клиенты шлют данные через webhook'и.
*   **Прямые подключения к турнирным операторам** — ESL, BLAST, PGL имеют свои API для аккредитованных аналитических платформ.

В проект нужно добавить:

```
apps/data-ingestors/
  ├── gsi-adapter/       # Парсер пакетов Game State Integration
  ├── steam-adapter/     # Коннектор к Steam API
  └── webhook-receiver/  # Приемник webhook'ов от партнеров
```

### Слой 2: Enrichment (обогащение данных)

Голые события типа "герой X убил героя Y" — это скучно. В реальном продакшене данные нужно обогащать:

*   Кто этот игрок? Его MMR, история, винрейт на этом герое?
*   Какая турнирная ситуация? Это групповой этап или финал?
*   Исторические данные: как команда играла этот драфт в последних 50 матчах?

Добавляем:
```
apps/enrichment-service/  # Сервис, который обогащает сырые события
  ├── player-profile-resolver.ts
  ├── tournament-context.ts
  └── historical-stats.ts
```

### Слой 3: Real-time Analytics (то, ради чего всё затевалось)

Твой `tracker-service` — это хорошо, но в реальности нужны конкретные аналитические пайплайны:

*   **Net Worth Tracker** — отслеживание золота команд в реальном времени с предиктивными кривыми.
*   **Power Spike Detector** — определение моментов силы героев (например, BKB куплен).
*   **Roshan Timer** — таймеры Рошана в Dota 2.
*   **Map Control Analyzer** — кто контролирует карту в CS2.
*   **Win Probability Engine** — математическая модель вероятности победы (не только AI, но и статистика).

Добавляем:
```
apps/analytics-pipelines/
  ├── net-worth-pipeline.ts
  ├── power-spike-detector.ts
  └── win-probability-engine.ts
```

### Слой 4: Multi-tenancy (работа с клиентами)

В реальности у тебя будут разные клиенты: букмекеры, стриминговые платформы, тренерские штабы. Каждому нужно разное:

*   Букмекеру — задержку не более 100мс и только коэффициенты.
*   Тренеру — полную детализацию и AI-инсайты.
*   Стримеру — красивый оверлей с инфографикой.

Добавляем:
```
packages/tenant-system/
  ├── tenant-resolution.ts      # Кто ты и что тебе можно
  ├── rate-limiting.ts          # Кому сколько запросов
  └── feature-flags.ts          # Какие фичи какому клиенту
```

### Слой 5: Real Production Concerns (то, о чем в ТЗ молчат)

**5.1 Authentication & Authorization**
В реальности нужен полноценный auth:

*   OAuth2/OIDC для пользователей
*   API Keys для B2B-клиентов
*   Rate limiting по ключам
*   WebSocket authentication (JWT в query string при коннекте)

**5.2 Circuit Breakers**
Если упал Steam API или Claude, твой сервис не должен упасть следом. Нужны паттерны отказоустойчивости:

*   Circuit Breaker на внешние вызовы
*   Graceful degradation (если AI недоступен, показываем статистический прогноз)
*   Retry policies с exponential backoff

**5.3 Data Archival & Partitioning**
Данные копятся бешеными темпами. Через месяц у тебя миллиарды записей. Нужно:

*   Партиционирование таблиц в PostgreSQL по датам/турнирам
*   Автоматическая архивация в S3-совместимое хранилище (MinIO)
*   Политики retention (например, сырые логи храним 7 дней, агрегаты — год)

**5.4 CQRS & Event Sourcing**
Вместо простых транзакций — полноценное событийное хранилище:

*   Все события матча — это event stream
*   Аналитические агрегаты строятся из событий
*   Можно перегенерировать любой отчет за прошлый период

**5.5 Health Check & Readiness Probes**
Kubernetes должен знать, жив ли твой сервис:

*   `/health` — сервис жив?
*   `/ready` — готов принимать трафик? (Redis доступен? PostgreSQL отвечает?)
*   `/metrics` — Prometheus-эндпоинт с метриками

### Слой 6: Observability (серьезный мониторинг)

ТЗ упоминает логирование, но в реальности нужно больше:

*   **Distributed Tracing** (OpenTelemetry + Jaeger) — чтобы видеть путь запроса через все микросервисы.
*   **Metrics** (Prometheus + Grafana) — сколько пакетов обработано, какая задержка, сколько дубликатов отбито.
*   **Alerting** (AlertManager) — если задержка > 200мс, слать алерт в Slack/PagerDuty.
*   **Synthetic Monitoring** — робот, который постоянно проверяет, работает ли система.

## Реалистичный Flow Данных в Продакшене

1.  **GSI/Steam API** → сырые пакеты прилетают в `data-ingestors`
2.  **data-ingestors** → валидируют, нормализуют и кладут в RabbitMQ (сырой стрим)
3.  **enrichment-service** → читает из RabbitMQ, обогащает, кладет обратно (enriched stream)
4.  **tracker-service** → читает enriched stream, фильтрует дубликаты через Redis
5.  **analytics-pipelines** → читают из tracker-service, считают метрики
6.  **AI Module** → получает контекст, генерирует инсайты
7.  **WebSocket Gateway** → рассылает всем подключенным клиентам
8.  **Client SDK** → получает данные, рендерит на UI

И всё это под мониторингом Prometheus, с трассировкой в Jaeger и алертами в PagerDuty.

**Ключевой момент:** Текущее ТЗ — это MVP (минимально жизнеспособный продукт). То, что описано — это Production-grade система. Между ними обычно 6-12 месяцев разработки командой из 3-5 человек. Но архитектурно закладывать эти вещи нужно с самого начала.
